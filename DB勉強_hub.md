## 失われた事実
**「その時、何が起きたか」という過去の事実が、上書きによって消えてしまわない設計になっているか？**
時間軸と直行しないデータ = 履歴を保存する
- <mark>行に有効期限</mark>を持たせる
	
- 履歴として持たせる<mark>(スナップショット)</mark>
	注文の価格などはマスター参照ではなく、注文テーブルに値をコピーして保持する。
- 履歴テーブルを作る
	Updateすることで過程が消えてしまう場合、
	「現在の状態」をもつテーブルと「状態の遷移過程」をもつテーブルでわける
	○○historyテーブルってこと。
	
また、パフォーマンス劣化などの観点からRDBの責務内で履歴の保持をしない場合、
以下の手段で必ず持たせるようにできているか。
- <mark>遅延レプリケーション</mark>の使用
- <mark>Elasticsearch</mark>などの分析ツールに保存

## やりすぎたJOIN
 **JOINの特性を考慮し、Indexの活用や不要なJOINの削減を行えているか？**

**■JOINの削減方法**
- <mark>Index</mark>を有効活用
	一意なIndexを貼ることで計算量の削減が期待できる
	
- アルゴリズムの考慮
	- [[Nested Loop Join]]
		- 1行ずつループして処理
		- 駆動表が小さいほど高速
		
	- [[Hash Join]]
		- 駆動表をHash化して結合
		- 十分なメモリが必要
		
	- [[Sort Merge Join]]
		- 表の大部分を結合する場合に有効
		- Indexが使用できる場合はHash Joinより早い場合もある
		
- <mark>View</mark>を活用したリファクタリング
	JOINの削減を狙えるうえ、クエリをシンプルにすることができる。
	
- <mark>マテリアライズド・ビュー</mark>に集計結果を保存する
	再作成のときにテーブルの作り直しが不要で、共有ロックのリフレッシュのみでいい。


JOINは計算量の掛け算と言われる。
<mark>テーブルスキャン</mark>の場合、1,000行と1,000行のJOINは1,000,000行となる。
JOINはSQLの中で最も重い処理のひとつであり、不用意にJOINをするとパフォーマンス劣化の原因となる。

## 効かないIndex
**Indexが有効に活用できるような設計となっているか？**

以下の場合はIndexが利用されない。
- 検索結果が多く、全体の件数が少ない
	Indexが有効なのは以下の場合
	1. 検索結果がテーブル全体の20%未満
	2. 検索対象のテーブルが十分に大きい
	
- 条件にIndexを設定した列を使っているか？
	WHERE句やON句で使用されないとIndexは利用されない。
	また、関数を使っている場合なども利用されない為注意が必要。
	
	❌ `WHERE FUNC(col) = ...` (Indexが死ぬ)
	⭕ `WHERE col = ...` (Indexが効く)
	
- カーディナリティが低い列にIndexを貼っていないか？
	B-Tree INDEXの場合、<mark>カーディナリティ</mark>が高い列にIndexを貼ることが有効。
	`性別`などよりも`メールアドレス`のような列に貼るべき。
	
- <mark>LIKE検索</mark>が前提の設計になっていないか？
	標準でIndexを利用するのは前方一致のみ
	
- <mark>統計情報</mark>と実際のテーブルで乖離がないか？
	次の場合に統計情報と実際のテーブルで乖離が発生する。
	1. サンプリングの前に大量のデータ更新が行われた
	2. サンプリングで偏ったデータを収集した

**※注意**
Indexを設定すると、そのたびにB-Treeを設定しなければならないため、
更新処理(CUD)が遅くなる<mark>(Indexショットガン)。</mark>

リリース初期など、データ傾向が見えない場合は、実際の傾向を見て判断することも大事。
(つまり、実行計画とかを見ずに脳死でINDEXを貼るのはよくないってこと。)

## フラグの闇

**<mark>「とりあえず削除フラグ」</mark>パターンに陥っていないか？**

データの物理削除(DELETE)を行うと、後から戻してほしいといった要望に対応できない為、
削除フラグを設定することがある。
しかし、以下のようなデメリットがある。

- クエリの複雑化
	
- <mark>UNIQUE制約</mark>が使えない
	
- カーディナリティが低くなる
	そのうえ、カーディナリティが低い(Indexが有効活用できない)列を毎回絞り込み条件に入れないといけない
	
- マルチカラムアトリビュートになる
	ある列が状態に依存している場合、におこる
	たとえば「削除理由」とかは削除しない限りNULL = マルチカラムアトリビュート

以下のような対応策がある
- <mark>状態ごとのテーブル分割</mark>を行う
	[[statusカラムを用意する前に考えたいことは]]
	
	削除済みのデータを移す操作はアプリケーション側で行うか、トリガーを使用する。
	ただ、<mark>「早すぎる最適化」</mark>にも注意

- <mark>View</mark>を使用する
	アプリケーション側からの参照はシンプルになる。
	しかし、高速化にはつながらない。

## ソートの依存
**ORDER BY句の振る舞いを理解し、パフォーマンスを意識できているか？**

エグゼキュータによる評価順序
1. FROM
2. ON
3. JOIN
4. WHERE
5. GROUP BY
6. HAVING
7. SELECT
8. DISTINCT
9. ORDER BY
10. LIMIT

ORDER BY句はSELECTの後に実行される。
つまり、データを取り出してからバラバラのデータを並び替えているため、高コストな処理。

対応策
- <mark>WHERE句狙いのINDEX</mark>
	事前にWHERE句で対象を絞り込むことでパフォーマンスを大きく向上させることができる。
	```sql
	SELECT * FROM users WHERE 性別 = '男性' ORDER BY 出身県id LIMIT 5
	```
	1. 性別のINDEXを利用して検索
	2. 該当の結果を取り出してそれをソート
	3. ソート中に5件が確定したらそれを返す

- <mark>ORDER BY句狙いのINDEX</mark>
	<mark>B-TreeIndex</mark>はデータをソート済みの状態で保存する。
	つまり、対象のソート結果とIndexが同じなら、Indexから取り出せばいい。
	```sql
	SELECT * FROM users WHERE 性別 = '男性' ORDER BY 出身県id LIMIT 5
	```
	↑出身県idにIndexを持たせる
	これにより、ソートの処理をすることなく、Indexから該当のデータを順番に取り出すだけとなる。

## 隠された状態
**データにビジネスロックを持たせていないか？**

- <mark>スマートカラム</mark>
	IDには`識別できる一意の値`以上の意味を持たせてはならない。
	例えば、IDの先頭が9なら管理者、8なら....
	みたいに、1つのカラムに複数の意味を持たせた場合、データの集計には関数を使用しなければならない。
	関数を使用するとIndexが使用できなくなるため、パフォーマンスの劣化につながる。

- 複数の目的に使われるテーブル
	複数の目的で使われるテーブルを作ってはならない。
	(例)ユーザーの属性として一般ユーザと管理者ユーザーを一つのテーブルとして持たせる
	
	データの属性によって
	- 入る値が変わるカラム
	- NULLが入るカラム
	がある場合は複数目的で使われている可能性がある。
	
	似たようなデータは一つのテーブルにしてしまいがちだが、
	パフォーマンスの観点からもテーブルを分けた方がいいケースが圧倒的に多い。
	
	似たアンチパターン
	- <mark>EAV(エンティティ・アトリビュート・バリュー)</mark>
		複数の目的に使われるカラムを用意する設計。
		スキーマレスな設計ができるが、デメリットの方が多い。
		レコード単位で状態を隠す手法
		
	- <mark>ポリモーフィック関連</mark>
		子テーブルが複数の親テーブルを持つ
		テーブル単位で状態を隠す手法
		→対応策として<mark>交差テーブルを用意</mark>するのが良い。
		
- 型(Type)の不一致
	日付を文字列で入れると範囲検索ができなくなる

## 強すぎる制約
**「早すぎる最適化」** をしていないか？
**「強すぎる制約」** が掛かっていないか？

| 段階     | 説明                                                                                                           |
| ------ | ------------------------------------------------------------------------------------------------------------ |
| 制約なし   | 制約が存在せず、自由にデータが入る                                                                                            |
| 弱い制約   | - NOT NULL制約<br>- UNIQUE制約<br>- 外部キー制約<br>など、データ構造を守るうえで最低限の制約が掛かった状態                                        |
| 強い制約   | - CHECK制約<br>- EXCLUDE制約<br>等が掛かった状態。<br>制約の内容は一般的な事実の範囲に収まり、守るべきデッドラインが明確になっていること。<br>(例えば、都道府県は47個しかない、など) |
| 強すぎる制約 | 制約の内容が「システムの仕様」や「ビジネスルール」にまで及んでいる                                                                            |

**<mark>制約</mark>** →データを守るためのルール
**<mark>規約</mark>** →バリデーションなど、アプリケーション側で担保すべきルール

多くのシステムは適切な正規化と弱い制約によって担保できる。
ビジネスロジックに伴うデータの制限は、規約によって担保する。

### 例
ぱっと見制約なのか規約なのか判断しかねる場合も存在する。
例えば以下の要件
> 1. 予約日時は未来のみ
> 2. 患者は同じ日時に複数の予約を持てない

小出はこれを見たときパッと判断できなかった。

結論、
1 → 規約
2 → 制約
となる。

| 観点    | 未来のみ    | 同時刻複数     |
| ----- | ------- | --------- |
| 時間依存  | あり（now） | なし        |
| 同時実行  | 関係なし    | 重大        |
| 例外    | 多い      | ほぼない（Lv1） |
| モデル前提 | 弱い      | 強い        |

**それが破られたときに業務データが破壊されないか？**
という観点をもつ。
1 → 破られたとしてもデータは破壊されない
2 → 同じ時刻に複数の予約が入る = 破壊的な業務データ

となる。


## 簡単すぎる不整合
**非正規化のリスクを考慮できているか**

- <mark>マルチカラムアトリビュート</mark>
	アンチパターン。
	あらかじめ複数列を用意しておき値があった場合にその列に値を格納し、値が無い場合はNULLのままにする。
	
	解決策
		値を「列」ではなく「行に」格納する
	
	詳しくは↓
	(https://qiita.com/fktnkit/items/52c45c61db0a209694ca)

- 外部キー制約などによってロックが発生している場合
	親テーブルに対して適切に<mark>共有ロック</mark>を使用することで防げる

- JOINのコストにより、パフォーマンスに問題が出ている場合
	正規化をすると値の重複が無くなるため、
	結果的にパフォーマンスが向上する場合が多い。
### 非正規化の代替案
1. CHECK制約
2. ENUM型
