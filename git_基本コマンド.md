#git 
### <u>git init</u>
#### コマンド
```markdown
- ディレクトリを指定
	git init [ディレクトリ名]

- 現在のディレクトリを指定
	git init
```

#### 概要
新しいリポジトリの作成を行う。
基本的に、プロジェクトの一番最初に一度だけ行う。

### <u>git status</u>

#### コマンド
```markdown
git status
```

#### 概要
- ブランチ名
- ステージングの状態
- ワークツリー
の変更状況を確認するためのコマンド。
つまり、<mark>変更ファイルをチェック</mark>するためのコマンド。
<mark>未追跡のコマンドが無いか</mark>の確認にもよく使われる

### <u>git diff</u>

### コマンド
```markdown
- リポジトリとワークツリー間の差異をチェック
	git diff

- リポジトリとステージの差分をチェック
	git diff --staged
```

#### 概要
ファイルの変更内容を確認

### <u>git add</u>

#### コマンド
```markdown
- 特定のファイルを登録
	git add [ファイル名]
  
- フォルダ内のすべての変更を登録
	git add .

- 変更や削除されたファイルのみを登録(新規作成ファイルは対象外)
	git add -u

- すべての変更、追加、削除をまとめて登録
	git add -A

- 変更箇所を対話的に選んで登録する
	-git add -p
 
```

#### 概要
変更したファイルや新規作成したファイルを、
次に行うコミットの「準備(ステージング)」として登録するコマンド。

ファイルを編集しただけではgitの管理下に置かれないため、
このコマンドを用いて記録対象を指定する。

#### 注意
コマンドを実行した後は、必ず`git status`コマンドで意図したファイルがステージングされているか確認すること。

### <u>git commit</u>

### コマンド
```markdown
git commit -m "[コミットコメント]"
```

#### 概要
ステージングされたファイルのスナップショットをローカルリポジトリに記録する操作。
コミットしたスナップショットは履歴として保存され、以前の状態を保持することができる。

### <u>git log</u>

#### コマンド
```markdown
git log
```

#### 概要
ローカルリポジトリの変更履歴を確認するコマンド。
コミットの履歴を確認するときに使用する。

#### <u>git branch</u>

#### コマンド
```markdown
- ブランチの一覧表示
	git branch

- 新しいブランチの作成
	git branch [ブランチ名]

- ブランチの削除
	git branch -d [ブランチ名]
```

#### 概要
ブランチの
- 一覧表示
- 作成
- 削除
- 名前変更
などを行うコマンド。
開発の履歴を分岐させ、メインの作業環境に影響を与えずに新機能の開発やバグ修正を行うことが出来る。

#### <u>git checkout</u>

#### コマンド
```markdown
- 既存のブランチへの切り替え
	git checkout [ブランチ名]

- 新しいブランチを作成して切り替え
	git checkout -b [ブランチ名]
```

#### 概要
- ブランチの切り替え
- ファイルの復元
- 新しいブランチの作成
を行うことができる。

<mark>多機能</mark>だが、後述する`git switch`と比べ、<mark>誤ってファイルを上書きするリスク</mark>などがある。

#### <u>git switch</u>

#### コマンド
```markdown
- 既存ブランチへの切り替え
	git switch　[ブランチ名]

- 新しいブランチを作成して切り替え
	git switch -c [ブランチ名]
```

#### 概要
主に
- ブランチの切り替え
- 作成
だけに特化している。

前述した`git checkout`に比べ、<mark>安全性が高く</mark>、誤ってファイルを上書きするリスクが低い。

### <u>git merge</u>

#### コマンド
```markdown
git merge [ブランチ名]
```

#### 概要
現在のブランチと任意のブランチを統合するためのコマンド。

### <u>git_push</u>

#### コマンド
```markdown
git push [url] [ブランチ名]
```

#### 概要
ローカル環境で行ったコミットをリモートリポジトリにアップロードして反映させるコマンド。

### <u>git pull</u>

#### コマンド
```markdown
- 完全形
git pull [url] [ブランチ名]

- 省略形
git pull
```

#### 概要
リモートリポジトリから最新の状態をローカルリポジトリに反映するコマンド。
後述する`fetch`と似たコマンドだが、こちらは`fetch`と`merge`を同時に行うことができる。
しかし、ワークツリーとリモートリポジトリの間でコンフリクトが起きた場合に戻すのが難しい(らしい？)

メインブランチにいるときは`pull`
その他のブランチにいるときは`fetch`

を使うと安全。

### <u>git_revert</u>

#### コマンド・手順
```markdown
# 1. mainブランチに切り替える 
	git checkout main 
	
# 2. コミット履歴を確認 
	git log --oneline 
	
# 3. 戻したいコミットを確認（例: 201016b） 
	git revert --no-commit 201016b..HEAD 
	
# 4. 1つのコミットとしてまとめる 
	git commit -m "Revert to 201016b" 
	
# 5. push
	git push origin main
	
```

##### 手順詳細
1. mainブランチに切り替え
```git
git checkout main
```

2. コミット履歴を1行で一覧表示
```git
git log --oneline
```

```git
//履歴が見れる 
a3c9b17 Fix typo in footer 
201016b Update hero section 
1f9e3a0 Add header navigation 
履歴のID（ハッシュ）をコピーしておきます。
```

3. 指定した範囲のコミットを「打ち消す」 準備をする(201016b のコミットの戻る例)
```
git revert --no-commit 201016b..HEAD
```

💡ポイント
- `201016b..HEAD` は「201016b以降のすべての変更を取り消す」範囲指定
- `--no-commit` を付けることで、自動コミットをせず、**まとめて1つのコミット**にできる。これを省略すると、コミットごとに「Revertコミット」が大量に作成されてしまう。

4. 打消し変更を1つの新しいコミットとして記録
```
git commit -m "Revert to 201016b"
```

5. mainブランチをpush
```
git push origin main
```

#### 概要
`revert`は、<mark>履歴を残したまま前のコミットの状態へ戻す</mark>ことができる。

💡ポイント
似たコマンドで`reset`があるが、こちらは履歴を消してしまうため、
チーム開発においては`revert`を使う方が安全。
