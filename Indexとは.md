# ■Indexとは
テーブル内の特定のレコード（行）へのポインタ
つまり、それを見れば<mark>指定した行を一発で特定できる</mark>

### 基本的な考え方
HDDの1[[ブロック]]に格納できるのは512バイト。
例えば1レコード128バイトなら、`512/128`で1ブロックにつき`4レコード`格納することができる

↓これで1ブロック

| 従業員ID(8byte) | 名前(50byte) | 備考(74byte) |
| ------------ | ---------- | ---------- |
| 1            | 小出         | xxx        |
| 2            | 佐藤         | xxx        |
| 3            | 羽田         | xxx        |
| 4            | 沖縄         | xxx        |
従業員が1000人いた場合は `250`個ブロックが作られることとなり、
その中から従業員ID = `10`を探そうと思ったら、
平均 `125`(250/2)個のブロックを探索しないといけなくなる。
これは効率が悪い。

キーとポインタを持つインデックステーブルを新たに作成する。

![[Pasted image 20251231122340.png]]
インデックステーブルは1レコード辺り`16byte`となるため、
1ブロックにつき`32`個格納することができる。
512 / 32で、16ブロックとなるため、
探索回数は平均8ブロックとなる。

これが基本的な考え方だが、さらに効率よくデータ検索を行うために
B-Treeインデックスを用いる。

# B-Treeインデックス

B-Treeは以下の特徴を持つ平行木

## B-Treeの特徴
### 1. ノードの中身：行き先案内板

> **<mark>各ノードは、キーと実データへのポインタ、子ノードへのポインタを持つ**</mark>

「ノード」とは、情報の入った**「箱」**のことです。

- **キー**：<mark>検索の目印（本のIDや学籍番号など）</mark>。
    
- **実データへのポインタ**：その本が置いてある「本棚の具体的な場所（住所）」へのメモ。
    
- **子ノードへのポインタ**：もっと詳しい情報が載っている<mark>「別の案内板」への矢印</mark>
    

### 2. 並び順：整理整頓のルール

> ノード内のキーは昇順にソートされる
> 
> <mark>親のキーは「左の子より大きく、右の子より小さい」</mark>

これは**「辞書」**をイメージしてください。「あ」の次は「い」と決まっているから、探しやすいですよね。

- 案内板に「50番」と書いてあったら、**左側の道**には「50番未満」のデータが、**右側の道**には「50番より大きい」データが置いてあります。
    
- このルールがあるおかげで、迷わずに最短距離でデータにたどり着けます。
    

### 3. リーフの高さ：全員同じスタートライン

> **<mark>リーフノード（末端）は全て同じ階層に存在する</mark>**

「リーフ（葉）」とは、一番下の階層にあるノードのことです。B-Treeでは、どのデータを探すにしても、必ず同じ回数だけ案内板をチェックすれば済むようになっています。

<mark>「あるデータはすぐ見つかるけど、あるデータは探すのに1時間かかる」という不公平が起きない</mark>、非常に平等な構造です。



### 4. 子どもの数とキーの数：部屋の定員ルール

> **<mark>1つのノードは、最小で t 個、最大で t 個の子ノードを持つ</mark>**

ここが一番難しく見えるかもしれませんが、要は**「部屋（ノード）の定員が決まっている」**ということです。

- **スカスカ禁止（下限 $t$）**：部屋が空きすぎると、案内板ばかり増えて効率が悪いので、ある程度は詰め込みます。
    
- **パンパン禁止（上限 $2t$）**：部屋が溢れそうになったら、部屋を2つに分割して、バランスを保ちます。
    

この<mark>「増えたら分ける、減ったら合体させる」</mark>というルールによって、前述の「同じ階層（高さ）」が維持されています。

---

#### 結局、何がすごいの？

普通の「二分探索木」だと、データが偏ったときに「めちゃくちゃ縦に長いツリー」になってしまうことがありますが、B-Treeは**「横にどっしり構えたピラミッド」**のような形をキープします。

これによって、ハードディスクのような「読み込みに時間がかかる場所」からデータを探す際、**<mark>「案内板を見る回数」を劇的に減らすことができる</mark>**のが最大のメリットです。





## いつ使うべきか?

### 大規模なテーブルに作成
データ量が少ないテーブルの場合、フルスキャンとB-Treeインデックスの処理速度に大きな差はありません。そのため、レコード数の少ないテーブル内のカラムにインデックスを作成しても効果はありません。<mark>レコード数が1万行以下の場合は、ほとんど効果がない</mark>ようです。

## カーディナリティの高い列に作成
**カーディナリティ**とは、あるテーブルのカラムが持つデータの種類の数です。

ここで、「性別」と「メールアドレス」の例を考えてみます。性別カラム内のデータは、「男性」、「女性」、「不明」の3種類になり、カーディナリティは3となり低いです。一方で、メールアドレスは重複が許されないためユーザーの数だけメールアドレスが存在することになります。よって、メールアドレスカラムのカーディナリティは高いです。

### **カーディナリティの高い列に作成する理由**  
まず、カーディナリティが低いカラムにインデックスを作成する場合を考えます。ここでは便宜上、「男性：1」、「女性：2」というデータを50%ずつ持つ性別カラムを考えます。

この場合、かなり大雑把に捉えると、B-Treeの左の部分木には1、右の部分木には2が配置されます。よって、`キー: 1`を探索する場合、B-Treeの左の部分木、つまり半分のデータを探索する必要があり、B-Treeの良さを活かすことができません。

逆に、メールアドレスのような重複がないカラムにB-Treeインデックスを作成すると、<mark>毎回の探索で探索範囲が減っていく</mark>ので、高速に探索を行うことができます。

**更新性能が劣化する**  
インデックスが作成されているカラムのデータを変更すると、B-Treeで保持しているデータも変更する必要があります。具体的には、B-Tree内の変更前のデータを削除し、変更後のデータを挿入する必要があります。これにより、テーブルにインデックスを作成すればするほど、そのテーブルへの更新処理が重くなってしまいます。

<mark>インデックス作成による検索・挿入・削除の高速化と、更新速度劣化のトレードオフ</mark>を意識して、インデックスを作成する必要があります。

**主キーなどの一意制約の列には作成不要**  
<mark>主キーや一意制約のカラムに対して、DBMSは自動的にB-Treeインデックスを作成</mark>しているため、重複してインデックスを作成する必要はありません。
